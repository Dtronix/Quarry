# Quarry: Runtime Builder Removal Plan

## 1. Overview

Removes the runtime fallback builder classes (`QueryBuilder`, `JoinedQueryBuilder`, `DeleteBuilder`, `UpdateBuilder`, `InsertBuilder`) and their supporting infrastructure (`SqlBuilder`, `SqlModificationBuilder`, `QueryState` mutation paths). All query execution flows through carrier-intercepted paths generated by the source generator.

**Prerequisite:** The batch insert API redesign (impl-plan-inserts.md) must be completed first, making all insert chains carrier-eligible.

**Two-phase approach:**

- **Phase 1:** Make all chains carrier-intercepted. For unanalyzable chains (tier 3 / RuntimeBuild), the generator emits interceptors where the terminal builds SQL at runtime using a lightweight path built into the carrier chain. Every method call is intercepted — no builder classes needed.
- **Phase 2:** Delete the runtime builder implementations and simplify `QueryExecutor`/`ModificationExecutor` to carrier-only paths.

---

## 2. Current Architecture

### Runtime Fallback Path (being removed)
```
User Code → EntityAccessor<T>.Where() → new QueryBuilder<T>(state)
  → QueryBuilder<T>.OrderBy() → new QueryBuilder<T>(state.WithOrderBy(...))
    → QueryBuilder<T>.ExecuteFetchAllAsync()
      → SqlBuilder.BuildSelectSql(state) — runtime SQL generation
        → QueryExecutor.ExecuteFetchAllAsync(state, reader, ct)
```

### Carrier-Intercepted Path (keeping)
```
User Code → [InterceptsLocation] Users_abc() → new Chain_0 { Ctx = ... }
  → [InterceptsLocation] Where_def() → Unsafe.As<Chain_0>(builder); return
    → [InterceptsLocation] ExecuteFetchAllAsync_ghi()
      → pre-built SQL string literal
        → QueryExecutor.ExecuteCarrierWithCommandAsync(cmd, reader, ct)
```

### After Removal
Every chain follows the intercepted path. Unanalyzable chains get a runtime-terminal interceptor that builds SQL in the terminal method instead of pre-building it.

## 3. Phase 1: Universal Carrier Interception

### 3.1 Runtime-Terminal Carriers

For chains the generator cannot fully analyze (tier 3 / RuntimeBuild), instead of falling back to runtime builders, emit interceptors for all sites with a **runtime-terminal** at the execution method.

The clause interceptors are no-ops (just cast and return). The terminal interceptor collects clause data from the carrier and builds SQL at runtime.

**Carrier structure for runtime-terminal chains:**

```csharp
file sealed class Chain_N : CarrierBase<User>
{
    // Clause data accumulated by interceptors:
    internal List<(string Sql, object[]? Params)>? WhereClauses;
    internal List<(string Sql, bool Desc)>? OrderByClauses;
    internal int? LimitValue;
    internal int? OffsetValue;
    // ... other clause accumulators
}
```

**Clause interceptors** for runtime-terminal chains store clause SQL fragments on the carrier instead of discarding them:

```csharp
public static IQueryBuilder<User> Where_abc(
    this IQueryBuilder<User> builder,
    Expression<Func<User, bool>> predicate)
{
    var __c = Unsafe.As<Chain_N>(builder);
    // Runtime expression evaluation to SQL:
    __c.WhereClauses ??= new();
    __c.WhereClauses.Add(ClauseEvaluator.EvaluateWhere(predicate));
    return Unsafe.As<IQueryBuilder<User>>(builder);
}
```

**Terminal interceptor** assembles SQL from accumulated clauses:

```csharp
public static Task<IReadOnlyList<User>> ExecuteFetchAllAsync_xyz(
    this IQueryBuilder<User> builder, CancellationToken ct = default)
{
    var __c = Unsafe.As<Chain_N>(builder);
    var sql = RuntimeSqlAssembler.BuildSelect(
        "users", null, SqlDialect.SQLite,
        __c.WhereClauses, __c.OrderByClauses,
        __c.LimitValue, __c.OffsetValue, ...);
    var cmd = __c.Ctx!.Connection.CreateCommand();
    cmd.CommandText = sql;
    // bind params from accumulated clauses
    return QueryExecutor.ExecuteCarrierWithCommandAsync<User>(..., cmd, reader, ct);
}
```

### 3.2 RuntimeSqlAssembler

New lightweight static class in `src/Quarry/Internal/RuntimeSqlAssembler.cs`. Replaces `SqlBuilder` for the runtime-terminal path. Much simpler than the current `SqlBuilder` because clause SQL fragments are already translated — it just concatenates them with proper syntax.

```csharp
internal static class RuntimeSqlAssembler
{
    public static string BuildSelect(
        string tableName,
        string? schemaName,
        SqlDialect dialect,
        IReadOnlyList<(string Sql, object[]? Params)>? whereClauses,
        IReadOnlyList<(string Sql, bool Desc)>? orderByClauses,
        int? limit,
        int? offset,
        string? selectColumns,
        IReadOnlyList<JoinFragment>? joins,
        bool isDistinct)

    public static string BuildUpdate(
        string tableName, string? schemaName, SqlDialect dialect,
        IReadOnlyList<string> setClauses,
        IReadOnlyList<(string Sql, object[]? Params)>? whereClauses)

    public static string BuildDelete(
        string tableName, string? schemaName, SqlDialect dialect,
        IReadOnlyList<(string Sql, object[]? Params)>? whereClauses)
}
```

This does NOT evaluate expressions — the clause interceptors do that. `RuntimeSqlAssembler` only concatenates pre-evaluated SQL fragments with `WHERE`, `ORDER BY`, `LIMIT`, `OFFSET`, etc. keywords and handles dialect-specific syntax (SqlServer OFFSET/FETCH, parameter format).

### 3.3 ClauseEvaluator

New static class in `src/Quarry/Internal/ClauseEvaluator.cs`. Evaluates LINQ expressions to SQL fragments at runtime. This is a simplified subset of what the old `ExpressionSyntaxTranslator` did, but operating on `System.Linq.Expressions` at runtime rather than Roslyn syntax trees at compile time.

```csharp
internal static class ClauseEvaluator
{
    public static (string Sql, object[]? Params) EvaluateWhere<T>(
        Expression<Func<T, bool>> predicate, SqlDialect dialect)

    public static (string Sql, bool IsDescending) EvaluateOrderBy<T, TKey>(
        Expression<Func<T, TKey>> keySelector, SqlDialect dialect)

    public static string EvaluateSelect<T, TResult>(
        Func<T, TResult> selector)
}
```

Only needed for tier 3 chains. Tier 1 (PrebuiltDispatch) and tier 2 (PrequotedFragments) chains have all SQL pre-built and don't use `ClauseEvaluator`.

### 3.4 Generator Changes for Runtime-Terminal Emission

**ChainAnalyzer:** No changes. Already classifies chains as RuntimeBuild when they can't be analyzed.

**CarrierAnalyzer.AnalyzeNew:** RuntimeBuild chains become carrier-eligible with a special `CarrierKind.RuntimeTerminal` flag. The carrier has clause accumulator fields instead of pre-computed parameter fields.

**FileEmitter:** New emission path for RuntimeBuild chains:
- Emit carrier class with clause accumulator fields
- Emit clause interceptors that evaluate expressions and store results on carrier
- Emit terminal interceptor that calls `RuntimeSqlAssembler`

**CarrierEmitter:** New `EmitRuntimeTerminalCarrier` method for the runtime-terminal carrier class shape.

### 3.5 Eliminating EntityAccessor<T> as Runtime Factory

Currently, `EntityAccessor<T>` creates `QueryBuilder<T>` instances when methods are called without interception. After this change, `EntityAccessor<T>` becomes a zero-allocation struct that only exists as a type for the carrier to implement. All its methods throw (same as carrier bases) because they're always intercepted.

The generated `Users()` method on the context returns a carrier cast as `IEntityAccessor<T>`, not an `EntityAccessor<T>`.

## 4. Phase 2: Delete Runtime Builders

After Phase 1 is stable and all tests pass with universal carrier interception.

### 4.1 Files to Delete

| File | Lines | What It Was |
|------|-------|-------------|
| `src/Quarry/Query/QueryBuilder.cs` | 1,074 | Mutable SELECT query builder |
| `src/Quarry/Query/JoinedQueryBuilder.cs` | 1,095 | Mutable JOIN query builder |
| `src/Quarry/Query/Modification/DeleteBuilder.cs` | ~200 | DELETE builder |
| `src/Quarry/Query/Modification/UpdateBuilder.cs` | ~200 | UPDATE builder |
| `src/Quarry/Query/Modification/InsertBuilder.cs` | 118 | INSERT builder (single-entity path retained if needed, or moved to carrier) |
| `src/Quarry/Query/Modification/SqlModificationBuilder.cs` | ~120 | Runtime SQL generation for INSERT/UPDATE/DELETE |
| `src/Quarry/Query/SqlBuilder.cs` | ~300 | Runtime SQL generation for SELECT |
| **Total** | **~3,100** | |

### 4.2 Files to Simplify

**QueryExecutor.cs:** Remove runtime execution methods that take `QueryState`:
- Remove `ExecuteFetchAllAsync(QueryState, reader, ct)` and similar overloads
- Keep `ExecuteCarrierWithCommandAsync` and `ExecuteCarrierNonQueryWithCommandAsync`
- Keep shared helpers (parameter binding, reader utilities, logging)

**ModificationExecutor.cs:** Remove runtime methods that take `InsertState`/`UpdateState`/`DeleteState`:
- Remove `ExecuteInsertNonQueryAsync(InsertState, entities, ct)`
- Keep carrier execution methods

**EntityAccessor<T>:** Convert from factory class to pure throw-stub struct (matching carrier base pattern). Or delete entirely if carriers implement `IEntityAccessor<T>` through their base classes.

### 4.3 Files to Keep

| File | Why |
|------|-----|
| `CarrierBase<T>` | Carrier inheritance base |
| `JoinedCarrierBase<T1,T2>` (and 3, 4 variants) | Join carrier bases |
| `ModificationCarrierBase.cs` (DeleteCarrierBase, UpdateCarrierBase, InsertCarrierBase) | Modification carrier bases |
| `BatchInsertCarrierBase<T>` | New batch insert carrier base |
| `QueryExecutor.cs` (carrier methods only) | Shared execution logic |
| `ModificationExecutor.cs` (carrier methods only) | Shared modification execution |
| `RuntimeSqlAssembler.cs` | Lightweight SQL assembly for tier 3 chains |
| `ClauseEvaluator.cs` | Expression evaluation for tier 3 chains |
| `BatchInsertSqlBuilder.cs` | SQL expansion for batch inserts |
| All interfaces | Public API contract |

### 4.4 QueryState/InsertState/UpdateState/DeleteState

These are mutable state holders used by runtime builders. After Phase 2:

- `QueryState`: Remove or reduce to a minimal struct holding only what `RuntimeSqlAssembler` needs (dialect, table name, schema). The clause data lives on the carrier instead.
- `InsertState`: Remove. Column metadata comes from compile-time `InsertInfo`. Runtime data is on the carrier.
- `UpdateState`: Remove. SET clauses stored on carrier accumulators.
- `DeleteState`: Remove. WHERE clauses stored on carrier accumulators.

### 5.2 Coverage Gaps to Migrate Before Deletion

These SQL edge cases are only covered by runtime builder tests and need cross-dialect equivalents:

**1. SqlServer pagination — ORDER BY (SELECT NULL) injection**
When Limit is specified without OrderBy, SqlServer requires injecting `ORDER BY (SELECT NULL)` for OFFSET/FETCH syntax. Add to `CrossDialectMiscTests`:

```csharp
// Tests: Limit without OrderBy on SqlServer emits ORDER BY (SELECT NULL)
// Tests: Limit with OrderBy on SqlServer does NOT inject ORDER BY (SELECT NULL)
// Tests: Offset without Limit on SqlServer
```

**2. All-row operations (DELETE/UPDATE without WHERE)**
`.Delete().All().ExecuteNonQueryAsync()` and `.Update().All().Set(...).ExecuteNonQueryAsync()` — dangerous operations that skip WHERE. Add to `CrossDialectDeleteTests` and `CrossDialectUpdateTests`:

```csharp
// Tests: Delete().All() generates DELETE FROM table (no WHERE)
// Tests: Update().All().Set(...) generates UPDATE table SET ... (no WHERE)
```

**3. Schema qualification for all operation types**
Verify INSERT, UPDATE, DELETE with schema-qualified table names across all dialects. Audit `CrossDialectSchemaTests` and add missing operations.

**4. Subquery IN/NOT IN**
Cross-dialect tests only cover `.Any()`/`.Count()` (EXISTS pattern). Add `.Contains()` pattern that generates `IN (SELECT ...)`. Add to `CrossDialectSubqueryTests`.

**5. Batch insert row grouping per dialect**
Multi-row VALUES clause parameter formatting. This will be covered by the new batch insert API tests (impl-plan-inserts.md).

### 5.3 Tests to Keep Unchanged (~480 tests)

All cross-dialect, integration, generator, and interceptor tests continue to work. They test the carrier-intercepted path which is unchanged.

## 6. ToSql() and ToDiagnostics() Strategy

After runtime removal, `ToSql()` and `ToDiagnostics()` only work through intercepted paths. The generator always intercepts these methods on every chain, so this is transparent to users.

For tier 1 chains: `ToDiagnostics()` returns the pre-built SQL string literal.
For tier 3 chains: `ToDiagnostics()` terminal interceptor evaluates accumulated clauses via `RuntimeSqlAssembler` and returns the assembled SQL.

The runtime `ToSql()` on `QueryBuilder<T>` (which called `SqlBuilder.BuildSelectSql`) is deleted. If a user somehow calls `ToSql()` without interception, the carrier base throws — same as any other non-intercepted method.

## 7. Risk Mitigation

### Rollback safety
Phase 1 is additive — runtime builders still exist, they're just not used. If universal interception has issues, chains can still fall through to runtime builders during development.

### Incremental validation
Run the full 2,954-test suite after each step:
1. After adding `RuntimeSqlAssembler` and `ClauseEvaluator`
2. After updating generator to emit runtime-terminal carriers
3. After migrating coverage gap tests to cross-dialect
4. After deleting runtime builder files

### Monitoring for missed chains
Add a `#if DEBUG` assertion in carrier base throw stubs that logs which method was called without interception. This catches any chain the generator misses during development, before it reaches production.

## 8. Implementation Order

1. Migrate coverage gap tests to cross-dialect (5 test additions)
2. Implement `RuntimeSqlAssembler` and `ClauseEvaluator`
3. Update `CarrierAnalyzer` to make RuntimeBuild chains carrier-eligible
4. Update generator emission for runtime-terminal carriers
5. Verify all 2,954 tests pass with universal interception

8. Delete `QueryState`/`InsertState`/`UpdateState`/`DeleteState`
9. Remove ~348 runtime builder tests
10. Final test suite validation
