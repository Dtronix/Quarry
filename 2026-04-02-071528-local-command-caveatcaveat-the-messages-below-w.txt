  The Single-Pass Scanner Concept

  Instead of N string.Replace() calls, a single left-to-right scan that
  recognizes tokens and writes the output, maintaining a running offset:

  Input:  ...IN ({__COL_P0__}) AND (@p1 IS NULL OR "CreatedAt" >= @p2) AND (NOT
  (@p3)...
           ↑ offset=0          ↑ offset += (3-1) = 2
  Output: ...IN (@p0, @p1, @p2) AND (@p3 IS NULL OR "CreatedAt" >= @p4) AND (NOT
   (@p5)...

  Core algorithm:

  offset = 0
  scan left to right:
    "{__COL_P" → parse index N, look up size S
                  emit S formatted params starting at (N + offset)
                  offset += (S - 1)
                  also fill __col{N}Parts[] with the emitted names

    "@p" + digits → parse index N           (SQLite/SS)
                    emit "@p" + (N + offset)

    "$" + digits  → parse index N           (PostgreSQL)
                    emit "$" + (N + offset)

    anything else → copy verbatim

  Why it works: collections always appear LEFT of the scalars they'd collide
  with. The offset accumulates naturally as we scan forward. Each collection
  adds (size - 1) to the offset, which shifts all subsequent scalar indices.

  Edge cases

  ┌──────────────────┬──────────────────────────────────────────────────────┐
  │       Case       │                       Behavior                       │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ Empty collection │ Emit SELECT 1 WHERE 1=0, offset += (0-1) = -1.       │
  │  (size=0)        │ Scalars shift LEFT by 1 — correct since the          │
  │                  │ collection's original 1 slot is now gone             │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ Multiple         │ Each one adds its own delta. Scanner handles         │
  │ collections      │ naturally left-to-right                              │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ Conditionals     │ Each variant has its own SQL string. Scanner runs    │
  │ (per-variant     │ per-variant. Scalars present in that variant have    │
  │ SQL)             │ correct compile-time indices; the scanner just       │
  │                  │ shifts them                                          │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ MySQL ?          │ No digits to rewrite — just expand {__COL_P0__} to   │
  │                  │ ?, ?, ?. No offset tracking needed                   │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ @p inside quoted │ Only match @p + DIGITS. "approvedBy" won't match     │
  │  identifiers     │ since @p isn't followed by a digit                   │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ Multi-digit      │ Parse ALL trailing digits after @p / $               │
  │ indices (@p10)   │                                                      │
  └──────────────────┴──────────────────────────────────────────────────────┘

  Where does it live?

  Two options:

  A. Runtime helper in Quarry.Internal (like CollectionHelper.Materialize):
  // Called from generated code
  internal static class ParameterExpander
  {
      public static string Expand(
          string sql, SqlDialect dialect,
          Span<(int globalIndex, int size, string[] parts)> collections)
      { ... }
  }
  Generated code becomes one call:
  sql = ParameterExpander.Expand(sql, dialect,
      stackalloc[] { (0, __col0Len, __col0Parts) });

  B. Emitted inline by the source generator — the scanner loop is emitted as
  generated C# in each carrier. More code per carrier but no runtime dependency.

  Parameter binding side

  The binding code also needs shifted names for scalars. Two options:

  B1. Compute shift separately (simplest):
  var __shift = __col0Len - 1;  // emitted per collection
  // Binding:
  __p1.ParameterName = "@p" + (1 + __shift);  // runtime concat
  __p2.ParameterName = "@p" + (2 + __shift);

  B2. Scanner outputs a name array — the helper returns both rewritten SQL and a
   string[] of all parameter names in order. Binding indexes into it.

  Performance comparison

  ┌────────────────────────────┬──────────────────┬─────────────────────────┐
  │          Approach          │ SQL rewrite cost │  Scalar binding name    │
  │                            │                  │          cost           │
  ├────────────────────────────┼──────────────────┼─────────────────────────┤
  │ Single-pass scanner        │ O(n) one pass    │ shift arithmetic        │
  │                            │                  │ (trivial)               │
  ├────────────────────────────┼──────────────────┼─────────────────────────┤
  │ Separate namespace         │ O(n) one Replace │ Zero (compile-time      │
  │ (SQLite/SS)                │                  │ const)                  │
  ├────────────────────────────┼──────────────────┼─────────────────────────┤
  │ Scalar tokenization        │ O(n × k) k       │ k string concats        │
  │                            │ Replaces         │                         │
  └────────────────────────────┴──────────────────┴─────────────────────────┘

  The scanner is O(n) regardless of parameter count. Separate namespace is also
  O(n) for SQLite/SS but can't handle PG. Scalar tokenization is O(n×k).

  Feasibility verdict

  Definitely feasible. The scanner is straightforward — the SQL is generated
  (predictable format), so we don't need a full SQL parser. The main trade-off:

  - Pro: Best asymptotic perf, uniform across all dialects, clean sequential
  @p0..@pN output, solves PG naturally
  - Pro: Can fill __colNParts[] during the same scan (binding names for free)
  - Con: Slightly more complex than "just change the prefix" for SQLite/SS
  - Con: Runtime helper adds a method to Quarry.Internal (but CollectionHelper
  already sets this precedent)

  The biggest win: it eliminates the need for separate dialect code paths in the
   expansion logic. One scanner handles @pN, $N, and ? with a dialect switch on
  the token format.

  - Con: Queries with many conditional variants could generate a lot of code

  For the common case (1-4 variants), the inline builder is compact. For extreme
   cases, we could fall back to a segment array approach.

  }

  var sql = __c.BuildCollectionSql();
  // Binding uses __c.__col0Parts and __c.__colShift

  Advantages:
  - Multi-variant switch logic is contained inside the method
  - Terminal method stays clean
  - Collection parts + shift live on the carrier — naturally scoped
  - Debuggable: you can inspect the carrier after the call

  Only added to carriers that actually have collections — zero cost for the
  common non-collection path.
