## Plan Compliance

| Finding | Severity | Why It Matters |
|---|---|---|
| QRY073 descriptor is defined and `BuildProjection` does emit a `DiagnosticInfo("QRY073", ŌĆ”)` on bind failure, but the descriptor is **not registered in `QuarryGenerator.s_deferredDescriptors`** (file `src/Quarry.Generator/QuarryGenerator.cs:754-772`). The two reporting loops (`QuarryGenerator.cs:524` and `:735`) call `GetDescriptorById(diag.DiagnosticId)` and silently `continue` when it returns null. Net effect: every call to `EmitProjectionSubqueryDiagnostic` is dropped; the user never sees QRY073. The plan explicitly justified the new diagnostic as "the silent (?)r.GetValue(N) regression cannot recur" ŌĆö it can. | High | Plan's stated failure-mode contract not delivered. The diagnostic exists in code but is unreachable end-to-end, so the silent-failure regression the fix was supposed to prevent will still occur for unresolvable navigations. |
| Phase 5 of the plan called for an end-to-end QRY073 emission test ("Negative test: `Select_Many_Sum_UnknownNavigation_RaisesDiagnostic`"). Direct consequence of this omission: nobody noticed the missing `s_deferredDescriptors` registration. | Medium | The deferred negative test would have caught the registration gap above. | Plan said the renderer should be invoked with `useGenericParamFormat: false` and `paramBase: 0`. ## Correctness

| Finding | Severity | Why It Matters |
|---|---|---|
| `ProjectedColumn.Equals` (line 337) uses `ReferenceEquals(SubqueryExpression, other.SubqueryExpression)` for the new field, while every other `SqlExpr` in the codebase has structural `DeepEquals`/hash. Two parses of the same source produce two distinct `SubqueryExpr` instances that are structurally equal but reference-distinct, so `ProjectedColumn` instances would compare unequal across incremental-generator runs even when the source hasn't changed. This can cause unnecessary cache invalidation in the source-generator pipeline. | Low | `ProjectedColumn` is `internal sealed record` and is part of `ProjectionInfo` cached by Roslyn; over-strict equality breaks incrementality but doesn't produce wrong output. |
| `ProjectionInfo.Equals` (line 117-130) was *not* updated to compare the new `LambdaParameterNames` field, and `GetHashCode` doesn't include it either. Two projections that differ only in lambda parameter name (e.g., `u =>` vs `x =>`) would be considered equal even though the `BuildProjection` bind step uses these names to look up entities, so a stale cache entry could bind a subquery against a parameter name that no longer exists in the current source. | Low | Same Roslyn-incrementality scenario; in practice changing a lambda parameter name also changes other things (column references), but the equality contract on ProjectionInfo is now violated. |
| The `IsResolved` check after `Bind` (`ChainAnalyzer.cs:1808-1812`) only inspects the top-level `boundExpr`. If the subquery's selector or predicate contains an unresolved sub-node (e.g., `NavigationAccessExpr` not bound, `CapturedValueExpr`), `Render` will emit a `/* unresolved ŌĆ” */` comment inline, producing SQL like `SELECT SUM(/* unresolved navigation: ŌĆ” */)` that quietly fails at runtime. The current check catches only "navigation property not on outer entity" failures. | Low | Defensive depth: a more thorough check would scan the rendered text for `/* unresolved` (or walk the bound tree). For #257's test surface this never triggers. |
| `TryResolveSelectorClrType` accepts only `ColumnRefExpr` selectors. For any computed selector (`o.Total + 1`, `o.Total * o.Quantity`, conditional, casts) it returns null, falling back to `"decimal"` for Sum / `"object"` for Min/Max. `Min(o => o.Total + 1)` would emit `ClrType = "object"` and a `GetValue` reader, producing a boxed `decimal` instead of a strongly-typed value. | Low | Aligns with what the existing Sql.* aggregate path does for unresolved types, but is a known limitation worth a comment. |
| `IsNavigationAggregateCall` requires `outer.Expression is MemberAccessExpressionSyntax` ŌĆö so `u.Orders.Where(o => ŌĆ”).Sum(o => o.Total)` (chained on an invocation) is silently not detected and falls through to the original broken path. Out of scope for the issue but the silent failure mode remains for that shape. | Low | Pre-existing gap; not introduced here. Worth surfacing because the diagnostic doesn't fire either (see Plan Compliance row 1). |
| The two Render callers (`SqlAssembler.cs:1080` and `ReaderCodeGenerator.cs:38, 328`) re-pass the rendered SQL through `SqlFormatting.QuoteSqlExpression`, which is a no-op when the string contains no `{ŌĆ”}` placeholders. The new code emits dialect-final SQL with already-quoted identifiers, which goes through cleanly ŌĆö verified by the cross-dialect SQL assertions. No defect, just confirming the existing pipeline contract isn't violated. | Low | Documentation-only observation; behavior is correct. |

## Security

No concerns.

The new code adds no new sources of user-controlled SQL text. Subquery SQL is built entirely from the parsed `SqlExpr` IR (column/property names from the source code's lambda) and dialect-specific `QuoteIdentifier` calls ŌĆö no string interpolation of user values, no `Sql.Raw`-style escape hatches. Captured-value flow (`ParamSlotExpr`) goes through the existing parameter-placeholder formatter.

## Codebase Consistency

| Finding | Severity | Why It Matters |
|---|---|---|
| `ProjectionInfo.Equals` does not include `LambdaParameterNames`, but every other init-only field on the class is included.
