## Correctness

| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| `ChainAnalyzer.ExtractShortTypeName` does not strip the `global::` prefix, while `TransitionBodyEmitter.ExtractDtoShortName` does. For a DTO declared in the global namespace, `RawCallSite.CteEntityTypeName` is set from `dtoType.ToFullyQualifiedDisplayString()` (`UsageSiteDiscovery.cs:3564`) which returns `"global::Foo"`. ChainAnalyzer's helper has no `.` in that string and returns `"global::Foo"` verbatim, so `cteDef.Name == "global::Foo"`. The emitter helper strips `global::` and produces `"Foo"`, so `cteDef.Name != siteCteName` is always true â†’ captured-param copy loop silently `break`s on the `cteDef.InnerPlan == null \|\| ... break;` path. Net effect for global-namespace DTOs: same silent runtime bug the b02a13e fix was supposed to close (P0..PN read as default). Also affects WITH clause rendering: `SqlAssembler.cs:159` would emit `WITH "global::Foo" AS (...)`, which most engines accept but is ugly. `src/Quarry.Generator/Parsing/ChainAnalyzer.cs:2037`, `src/Quarry.Generator/CodeGen/TransitionBodyEmitter.cs:150`. | Medium | Edge case (most users namespace their DTOs) but it's exactly the silent-failure mode the recent fix targeted. The two helpers should be unified (or `ExtractShortTypeName` should also strip `global::`). |
| `EntityRef` foreign-key detection in `ProjectionAnalyzer.BuildColumnInfoFromTypeSymbol` matches by type *name* only: `entityRefType.Name == "EntityRef" && TypeArguments.Length == 2`. There is no namespace check for `Quarry`. A user DTO referencing a same-named generic struct from another library/namespace (`MyLib.EntityRef<X, Y>`) would be incorrectly classified as a foreign key with `referencedEntityName = X.Name`, then downstream consumers (`ReaderCodeGenerator.cs:206`, `RawSqlBodyEmitter.cs:268`, `TerminalEmitHelpers.cs:597`) would attempt to construct `EntityRef<X, TKey>` from a key column. Tightening to `entityRefType.ContainingNamespace?.Name == "Quarry"` (or `OriginalDefinition` equality against the real Quarry type) would close the gap. `src/Quarry.Generator/Projection/ProjectionAnalyzer.cs:927`. | Low | Theoretical clash; very unlikely in practice. Worth flagging because the fix swapped a name-only heuristic for another (slightly narrower) name-only heuristic. |
| The new `EntityRef` FK detection does not handle nullable `EntityRef<X,Y>?`. A property typed as `EntityRef<User, int>?` (e.g. nullable navigation FK) would have `propertyType.Name == "Nullable"`, the type check would fail, and the property would not be flagged as a foreign key. The PRE-fix heuristic (name endsWith "Id") would have caught it (with the wrong null `referencedEntityName`); the new code silently treats it as `Standard`. `src/Quarry.Generator/Projection/ProjectionAnalyzer.cs:927`. | Low | Only impacts the discovery-time `AnalyzeFromTypeSymbol` projection path (Select on a type with no EntityInfo). Real entity definitions go through the fully-bound `Analyze(EntityInfo, ...)` path, which has its own FK metadata. Edge case but worth noting since the FK code path was just rewritten. |
| `CallSiteBinder` dialect-from-context fallback is gated by `!string.IsNullOrEmpty(raw.ContextClassName)`. For sites whose `ContextClassName` is null/empty (e.g. some non-CTE diagnostic-only sites where context resolution failed), the fallback short-circuits and `dialect` remains the placeholder `SqlDialect.PostgreSQL`. The change is strictly better than before for CTE chains, but it does NOT add a single-context fallback (the common case where there is exactly one context registered). Single-context fallback would close the residual hole. `src/Quarry.Generator/IR/CallSiteBinder.cs:60`. | Low | Pre-existing behavior preserved on the null-ContextClassName path; not introduced by this change. Filed for completeness because the surrounding code was just touched. |
| `EmitCteDefinition` matches a `cteDef` by name, then breaks on the *first* match without checking that the matched `cteDef.InnerPlan` actually corresponds to *this* With() site. With multi-CTE blocked by #206, the practical impact is zero today, but if a user writes `db.With<X>(a).With<X>(b)...` (same DTO type twice â€” illegal SQL but not yet rejected), the loop matches the first `cteDef` and copies its params for the SECOND With() too, masking the duplicate-CTE error. Cheap defense: emit a chain-analyzer diagnostic for duplicate CTE names. `src/Quarry.Generator/CodeGen/TransitionBodyEmitter.cs:123`. | Low | Edge case; #206 will need to address it anyway. Mentioning so it's not lost. |
| `cteInnerResults` is keyed by `argSpanStart` (`int`). If two `With(...)` invocations on different chains in the same source file happen to share the same span position (e.g., generated/templated code), entries collide. Realistically `argSpanStart` is unique per source position, so this only matters for synthetic input or obscure edge cases. `src/Quarry.Generator/Parsing/ChainAnalyzer.cs:132`. | Low | No real-world failure path; flagged because the keying scheme was newly introduced. |
