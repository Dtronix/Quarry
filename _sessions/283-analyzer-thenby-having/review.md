## Correctness
| Finding | Severity | Why It Matters |
|---|---|---|
| Anchor list `{ "OrderBy", "OrderByDescending" }` includes `OrderByDescending`, but the actual Quarry public API (`IEntityAccessor<T>.OrderBy`, `IQueryBuilder<T>.OrderBy`) has only a single `OrderBy<TKey>(..., Direction direction = Direction.Ascending)` overload. `OrderByDescending` exists only on `IOverClause` (window functions), which never appears in a chain that a `ThenBy` would target. The rule fires only when `Site.Kind == InterceptorKind.ThenBy`, and `UsageSiteDiscovery.InterceptableMethods` only maps `"ThenBy"` (not `"ThenByDescending"`) to that kind, so in practice the rule never sees a chain whose anchor would be `OrderByDescending` either. The defensive anchor entry is harmless dead code. | Low | Same applies to the code fix's `"ThenByDescending" => "OrderByDescending"` switch arm and the `ThenByDescendingWithoutOrderBy_Reports` / `ThenByToOrderByCodeFix_RegistersCodeFix_ThenByDescending` tests — the entire ThenByDescending pathway is never reachable on real Quarry chains because `UsageSiteDiscovery` does not register `"ThenByDescending"` as an interceptable method (so `Site.Kind` never equals `ThenBy` at such a call). The work doesn't break anything, but the plan's premise that "ThenBy / ThenByDescending" need to be handled symmetrically does not match the codebase's API surface (where descending sort is `Direction.Descending` argument, not a separate method name). |
| Code fix uses `WithName(SyntaxFactory.IdentifierName(newName))` which always produces an `IdentifierNameSyntax`. If a user wrote `.ThenBy<int>(x => x.Id)` (generic invocation), `memberAccess.Name` would be a `GenericNameSyntax` and the fix would silently strip the type-argument list. Quarry's `ThenBy` is generic (`ThenBy<TKey>`), and although call-side type arguments are rarely written explicitly, when they are present this loses them. The diagnostic location uses `outerMember.Name.GetLocation()` (which on a generic name covers the whole `ThenBy<int>` span), so the diagnostic still lands correctly, but the fix output would change `ThenBy<int>(...)` to `OrderBy(...)`. | Low | Realistic chain code rarely writes explicit type arguments (the `Direction` parameter is via overload, not generic), but the fix is still incorrect for the generic-name case. Not covered by tests. |
| Code fix path `node.AncestorsAndSelf().OfType<MemberAccessExpressionSyntax>().FirstOrDefault()` walks up from whatever `FindNode(diagnostic.Location.SourceSpan)` returns. The diagnostic location is on the `Name` of the outer member-access, so `FindNode` returns the `IdentifierNameSyntax` (or `GenericNameSyntax`) whose nearest member-access ancestor is exactly the desired outer one. This is correct, but if the diagnostic span ever drifted (e.g. someone reports on the whole invocation), the same lookup would still find the outermost member-access by `AncestorsAndSelf` ordering, which is the correct one. No defect; noting for completeness. | Nit | Defensively-shaped; behaves correctly under the current diagnostic location convention. |
| `HavingWithoutGroupByRule` uses `current = outerMember.Expression` *before* the loop (correct), then loops while `current is InvocationExpressionSyntax`. If the chain root before Having is `someExpr.Foo` (member access, not invocation — e.g. a stored property), the loop exits immediately without flagging. This matches the spec ("walk receiver chain of invocations"), but means the rule cannot see anchors hidden behind a property dereference like `cachedQuery.Having(...)` where `cachedQuery` was assigned `db.Users().GroupBy(...)`. The plan accepted this limitation implicitly (single-statement chain analysis only). Same applies to `ThenByWithoutOrderByRule`. | Nit | Aligns with peer rules which also only do single-statement chain walking. |

## Test Quality
| Finding | Severity | Why It Matters |
|---|---|---|
| Tests use synthetic `dynamic db` source so `CSharpSyntaxTree.ParseText` accepts them without compile errors. This bypasses semantic resolution entirely — the rule's `Site.Kind` is forcibly stamped via the helper's `kind` parameter. As a consequence, the tests never exercise `UsageSiteDiscovery.DiscoverRawCallSite`, so they don't validate that real Quarry chains would actually be classified as `InterceptorKind.ThenBy` / `InterceptorKind.Having`. This is consistent with how other PatternRule tests work, but it does mean the `ThenByDescending` test never validates whether `UsageSiteDiscovery` even routes a `ThenByDescending` call to `Kind.ThenBy` (it does not — see Correctness finding above). | Low | The end-to-end "real Quarry analyzer fires QRA403 on real Quarry source" path is untested, but no peer rules test that either. |
| `QRA403_NonThenBySite_NoReport` and `QRA404_NonHavingSite_NoReport` only test ONE non-matching kind (`OrderBy` and `Where` respectively). A more discerning test would exercise multiple kinds (e.g. `ExecuteFetchAll`, `Select`) to confirm short-circuit behavior across the range. Marginal value. | Nit | Pragma; current coverage suffices for a kind-equality guard. |
| No test verifies the diagnostic message format actually substitutes the method name token. `QRA403_ThenByDescendingWithoutOrderBy_Reports` only asserts `Has.Count.EqualTo(1)`; it does not check that the diagnostic message contains `"ThenByDescending"` (versus `"ThenBy"`). If someone swapped the `messageFormat` arg order or removed the `methodName` argument, the test wouldn't catch it. | Low | Consistent with other PatternRuleTests that also only count diagnostics; not a regression from baseline. |
| The "intervening-call" test for QRA404 (`QRA404_GroupByThenWhereThenHaving_NoReport`) uses `g.Count > 0` syntax inside `Where(g => g.Count > 0)` which would not normally be valid post-GroupBy on a real Quarry chain (GroupBy in Quarry doesn't surface a grouping where you'd then call `.Where(g => g.Count > 0)` as a transparent intervener), but since the source uses `dynamic db`, the parse succeeds. The test still validates the chain-walk transparency it intends to. | Nit | Source isn't realistic Quarry syntax, but the rule under test only walks method-name tokens, so realism doesn't matter here. |
| No test verifies that the code fix preserves the argument list verbatim — only the method name. A fix that, e.g., dropped the lambda argument would still satisfy `Does.Contain(".OrderBy(x => x.Id)")` because the substring is present. The functional test (`ThenByToOrderByCodeFix_AppliedFix_RewritesMethodName`) does check this string is contained, which catches the gross failure, but not subtler regressions (e.g. trivia loss, argument reordering). | Nit | Adequate for catching the common failures the fix could exhibit. |
| Three of the QRA403 tests exist for the OrderByDescending anchor and the ThenByDescending diagnostic, but as noted in Correctness these code paths are never exercised by real Quarry chains. The tests pass because the helper directly stamps `InterceptorKind.ThenBy` regardless of method name. | Low | Tests are not "wrong" but they validate behavior that is unreachable in production. |
