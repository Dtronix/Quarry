The only implementations are file-sealed carrier classes emitted by `Quarry.Generator` (`CarrierEmitter.cs:188-189` etc.). The `EntityAccessor<T>` type referenced in `InterceptorCodeGenerator.Utilities.cs:480` (`((EntityAccessor<{entityType}>)(object)builder).CreateQueryBuilder()`) is presumably a generator-emitted unbox helper; it is not declared in the runtime as a publicly-implemented type. No public concrete implementer is at risk. | Info | No struct-implements-interface gap. |
| Adding new default interface methods is binary-additive but a NEW public API surface for any external consumer of the `Quarry` package: callers who previously got CS1061 on `accessor.OrderBy(...)` will now successfully bind. There's no overload-resolution disambiguation risk for existing chains (`db.Users().OrderBy(...)` previously bound to `IQueryBuilder<T>.OrderBy` because `Users()` returned `IEntityAccessor<T>` and the only legal `OrderBy` was a hypothetical extension method or a misbinding — actually it was the old "chain transitions to IQueryBuilder via the first method" path). Existing tests passing (per workflow.md "3367 total, all green") indicates the generator pipeline still produces correct interceptors via the `IsEntityAccessorType` / `BuildReceiverType` path described in the plan, but the binding-target shift is a real semantic change consumers should be aware of. |
| The remarks update at `IEntityAccessor.cs:7-16` is a documentation breaking change for anyone who relied on the documented behavior "Chain-continuation methods only appear after the first clause" — the previous wording explicitly told users they had to call e.g. `.Where(...)` first. Source consumers reading the old docs may have built workarounds; those workarounds still work, but the docs no longer describe their necessity. | Info | Documentation change is intentional per plan. |
