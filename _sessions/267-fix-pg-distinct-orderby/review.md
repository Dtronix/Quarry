## Correctness

| Finding | Severity | Why It Matters |
|---|---|---|
| **`paramIndex` is not advanced after rendering extra ORDER BY expressions in the inner SELECT.** `SqlAssembler.cs:1502-1508` renders each `extraOrderExprs` entry with `SqlExprRenderer.Render(expr, dialect, paramIndex)` but does not add `CountParameters(expr)` to `paramIndex` between iterations or before WHERE rendering at line 1562. The plan explicitly contracts "those parameters are part of `plan.Parameters` and threaded through `paramIndex` exactly as in the flat path." Today the simple case is bare column refs (zero-param), so this is silent — but the moment a user writes `OrderBy(o => o.Total + capturedThreshold).Distinct().Select(...)`, the inner ORDER BY expr's parameter and the first WHERE/GROUP/HAVING parameter both render to the same `@p{paramIndex}` placeholder. On MySQL (`?` positional) the carrier provides values in chain-assigned order, so a fresh `?` slot is consumed twice — values would shift left by one for the rest of the query. On PG/SS/SQLite (named) two clauses claim `@p{N}` and one of them gets the wrong runtime value. | High | Latent correctness time-bomb. No test exercises ORDER BY with parameters under DISTINCT, so the regression escapes coverage. Fix is two lines: track paramsBefore in the loop and advance paramIndex; adjust `whereParamOffset`/`paramIndex` accordingly. |
| **Empty / entity-projection chains (`Distinct().Select(u => u)`) bypass the wrap.** `NeedsDistinctOrderByWrap` (`SqlAssembler.cs:1339`) early-returns false when `plan.Projection == null || plan.Projection.Columns.Count == 0`. If projection is empty the flat path emits `SELECT DISTINCT * ... ORDER BY <non-projected>`, which PG still rejects with 42P10. Verify whether `Select(u => u)` lowers to a populated `Projection.Columns` (one per entity column) — if it does, the wrap fires; if it lowers to an empty projection (with `*` rendered downstream), this is a gap. | Medium | If the entity-projection path ends with `Columns.Count == 0`, the original PG bug still exists for that shape. Worth either confirming via a regression test or removing the early-return and letting the wrap handle empty projection (printing all entity columns from `plan.PrimaryEntity` into the inner SELECT). |
| **CTE inner-chain re-render does not propagate the outer mask.** `SqlAssembler.cs:1429` (in the wrap path) calls `RenderSelectSql(cte.InnerPlan, mask: 0, dialect, ...)`. This matches the flat path's behavior at line 221 and is guarded by the same `Debug.Assert` for `PossibleMasks.Count <= 1`. Behavior parity with the flat path is preserved, but the assumption is now duplicated. If the assertion ever trips in the flat path it now also trips in the wrap, doubled. | Low | Not a regression — same behavior as flat. Worth noting because the assertion message ("Extend RenderSelectSql to render per outer mask if this triggers.") doesn't mention the second copy in the wrap. |
| `MayNeedDistinctOrderByWrap` (`SqlAssembler.cs:1378`) renders every ORDER BY expression with `parameterBaseIndex: 0` to compare for set membership. For dialects/exprs where the rendered string is parameter-position-sensitive (e.g., MySQL `?`), zero-param column refs are stable, but if an ORDER BY expression includes a parameter the rendered string for batch detection differs from the actual SQL emission position string, so detection still works (it's looking for "this expression's text" in the projection set, both rendered with paramOffset 0 — symmetric). Detection is therefore correct. | Low | Documenting the symmetry — both `NeedsDistinctOrderByWrap` and `RenderProjectionColumnRef` use paramOffset 0, so set-equality comparison is well-defined. |
| `outerOrderByAliases` is built from `activeOrderTerms`, but the outer ORDER BY emission loop (`SqlAssembler.cs:1617`) uses `activeOrderTerms.Count` and `outerOrderByAliases[i]` interchangeably. They're constructed in lockstep so the indices match. Verified. | Low | No defect; noting because the off-by-one risk was specifically called out in the task scope. |

## Security

No concerns.

The new identifiers (`__d`, `__c{i}`, `__o{i}`) are static literals with no user input. User-supplied column names and table aliases continue to flow through `SqlFormatting.QuoteIdentifier` which doubles the quote char to escape (`SqlFormatting.cs:44-49`), unchanged from the flat path. `RenderProjectionColumnRef` (`SqlAssembler.cs:1316-1323`) and the inner-projection emission (`SqlAssembler.cs:1490-1500`) both call `QuoteIdentifier` on the same fields the flat path quotes, so the existing escape contract carries over.
