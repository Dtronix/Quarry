## Correctness

| Finding | Severity | Why It Matters |
|---|---|---|
| `ChainAnalyzer` enriches aggregate columns whose type was unresolved at discovery time. After enrichment to `"int"`, the column is now an int-typed window aggregate without the flag. | low | Edge case — only triggers for `Min`/`Max OVER` against an int column when the entity property type wasn't yet resolved. Worth knowing the gating-once-at-discovery-time pattern leaks into this enrichment seam. |
| `RenderProjectionColumnRef` (`Z:/Projects/Quarry/274-ss-window-function-int/src/Quarry.Generator/IR/SqlAssembler.cs:1386-1391`) re-uses `AppendProjectionColumnSql`, so projection-column reference strings now carry the `CAST(... These strings are compared via string equality against ORDER BY expressions rendered through a separate path (`SqlExprRenderer.Render` at lines 1451 & 1476) for DISTINCT + ORDER-BY-on-non-projected detection. No existing test covers `Distinct() + window-function projection` on Ss to confirm the practical impact. | low-medium | Existing comment at line 1346 promises "byte-identical output between detection and emission" — the new symmetry is still maintained between the two `RenderProjectionColumnRef` callers and the inner emit (both wrap), but the symmetry between projection-ref-string and ORDER-BY-render-string is broken on Ss. |
