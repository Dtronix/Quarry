## Plan Compliance
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| Plan (workflow Decisions line 29) explicitly states "Emit QRY029 (existing diagnostic) via the projection pipeline when validation fails". Implementation does not surface QRY029 for projection-context `Sql.Raw` failures. `PipelineOrchestrator.cs:169-180` only emits QRY029 when `raw.Expression is RawCallExpr` (the Where-path IR expression). Projection analyzer builds a transient `RawCallExpr` for validation in `IsRawTemplateValid` and immediately discards it; the validation error is converted into a generic `ProjectionInfo.CreateFailed("Could not analyze tuple element at position N")` that produces a runtime-build chain fallback with no user-visible diagnostic. | medium | Where-path `Sql.Raw` with bad template → QRY029 (compile error). Select-path `Sql.Raw` with the same bad template → runtime-build chain, no compile-time signal. Plan note line 107 acknowledges scope-deferring this, but Decisions line 29 contradicts that. Documented drift. |
| Plan line 45 specifies `IsStaticField = cap.IsStaticField` be propagated from `CapturedValueExpr` to the emitted `ParameterInfo`. `AddCapturedAsProjectionParameter` copies `CapturedFieldName` and `CapturedFieldType` but never assigns `paramInfo.IsStaticCapture = captured.IsStaticField`. `SqlExprClauseTranslator.cs:92` (the canonical reference pattern) does propagate it. | low | Latent only — `SqlExprParser` never constructs a `CapturedValueExpr` with `IsStaticField=true` today (verified: no `WithIsStaticField` caller, no parser site sets it), and the walker-path only runs when `ResolveScalarArgSql` fast-path does NOT fire. The fast-path (used for captured locals/parameters) already sets `IsStaticCapture` correctly via its own logic. So this drift is correctness-neutral in practice but drifts from the plan and the `SqlExprClauseTranslator` pattern. |
| Plan line 20 describes walker handling `NestedProperty == "Id"` column refs (FK access like `u.Order.Id`). `ResolveColumnRefToPlaceholder` and `ResolveJoinedColumnRefToPlaceholder` both inspect only `PropertyName`, ignoring `NestedProperty`. | low | Behavior matches `GetColumnSql`/`GetJoinedColumnSql` (both also ignore `NestedProperty`). No test covers `Sql.Raw<int>("{0}", u.Order.Id)` in a projection; if the FK is stored under the navigation property name in `columnLookup`, the output is correct. If under the FK column name ("UserId"), the fallback emits `{Order}` which is wrong. Untested edge case. |

## Correctness
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| `RenderRawArgNode`'s `case SqlRawExpr raw: return raw.SqlText` emits raw text for any `SqlRawExpr` encountered. `SqlExprParser.ParseExpression` falls through to `new SqlRawExpr(expression.ToString())` as its default for **many** unhandled expression kinds (see `SqlExprParser.cs:130-131`). Notably, `ConditionalExpressionSyntax` (C# ternary) is parsed to `new SqlRawExpr("/* unsupported: C# ternary expression */")` (line 109). A user writing `Sql.Raw<int>("{0}", cond ? 1 : 2)` in a projection will emit the literal comment text as SQL. Other unhandled syntax nodes (e.g., method invocations not recognized by `ParseInvocation`, `ElementAccessExpressionSyntax`, `InterpolatedStringExpressionSyntax`) also fall through and emit C# source text verbatim into the SQL. | medium | Silent wrong-SQL emission — the very failure mode this PR fixes. Walker should treat `SqlRawExpr` as "unsupported" (return null) for projection arg rendering, since by definition `SqlRawExpr` is the parser's I-don't-know case. The Where-path IR does not have this problem because `SqlExprBinder`/`SqlExprRenderer` are driven by `RawCallExpr` validation and a more controlled parser path. |
| `TryExtractSqlRawTypeArg`'s fallback at line 2148 resolves the type from `methodSymbol.ReturnType` when the explicit type-arg path fails. The same fallback is used when the user calls `Sql.Raw` (non-generic, via implicit T inference) or when the arg-list parse is slightly unusual. If the method symbol itself is unresolved (TypeKind.Error on ReturnType), `BuildSqlRawInfo` returns `(null, null)` and the projection fails. This is stricter than aggregates' `"object"` fallback, which is intentional per the plan — but note that `clrType` is also consumed downstream by `ChainAnalyzer` to pick readers; if `GetSimpleTypeName(methodSymbol.ReturnType)` returns a non-intrinsic name (e.g., "T" for an unbound type parameter), that name flows through to the reader generator with no further validation. | low | Narrow edge case. Tests only cover intrinsic types (int/string/bool/DateTime). |
| `FormatLiteralForProjection`'s bool detection uses `literal.SqlText == "TRUE" || "true" || "1"`. `ParseLiteral` only ever emits `SqlText == "TRUE"` or `"FALSE"` for booleans (`SqlExprParser.cs:271,274`); the `"true"`/`"1"` branches are dead code unless some other upstream produces `LiteralExpr` with lowercase/1/0 text. Minor, but the dead checks add cognitive load. | info | Defensive but misleading — future readers may think there's a canonicalization issue. |
| `GetRawBinaryOperator`'s `_ => "?"` fallback silently emits `?` for any new operator added to `SqlBinaryOperator`. `?` is the MySQL parameter placeholder and will parse but produce unrelated runtime behavior. | low | Latent — all 16 current operators are handled. Future enum additions would silently drift. Throwing `ArgumentOutOfRangeException` would be consistent with fail-loud philosophy elsewhere in this PR. |
| The `HasStringOperand` string-concat guard rejects `Add` only when a direct operand is a string-typed `LiteralExpr` or `CapturedValueExpr`. It does NOT reject a string-typed `ColumnRefExpr` — because the walker doesn't know the column's CLR type (columnLookup lookup gives `ColumnInfo.ClrType`, but the guard only inspects the IR node). So `Sql.Raw<string>("{0}", u.Name + "_suffix")` is rejected (literal on right), but `Sql.Raw<string>("{0}", u.FirstName + u.LastName)` passes and emits `("FirstName" + "LastName")` — wrong on MySQL/SqlServer. | low | Guard is partial. Whichever result the PR wants, the partial guard is inconsistent. That said, `SqlExprRenderer` (the Where-path equivalent) has the same gap — the TODO at `SqlExprRenderer.cs:290-295` explicitly notes "Check if this could be string concatenation — if either side is a string column. This is a heuristic; the exact check would require type info". So the partial guard is consistent with the existing Where-path behavior rather than better. |
| `AddCapturedAsProjectionParameter` does not handle `expressionPath` — it copies `SyntaxText`, `VariableName`, and `ClrType` but ignores `CapturedValueExpr.ExpressionPath`. `SqlExprClauseTranslator.cs:90` and `ResolveScalarArgSql` both forward `expressionPath` to the `ParameterInfo` constructor. Absence may cause the captured-variable extractor to miss the deep-path capture signal for field-access chains like `obj.Inner.Field`. | low | Walker-path only fires when fast-path (ResolveScalarArgSql) rejects — which excludes bare identifiers and parameter-prefixed member accesses. Remaining walker captures are rare (function-call args, operator arms). Latent. |

## Codebase Consistency

Any future operator addition or dialect fix in `SqlExprRenderer` must be duplicated here. The code comment acknowledges this as a deliberate choice until `SqlExprRenderer` supports a canonical-projection output mode. | medium | Duplication is signposted but real. The fast-path/walker-path split in `RenderRawArgToCanonical` means both construction paths coexist. | low | Two code paths building the same entity; minor drift risk (see IsStaticCapture drop and expressionPath drop above). |
| `case "Raw" when semanticModel != null:` guard in `GetJoinedAggregateInfo` is inconsistent with `case "Raw":` in `GetAggregateInfo`. The guard is load-bearing (joined signature accepts `SemanticModel?`) but the defensive style differs from single-entity. | info | Minor style nit, functionally correct. |
