## Plan Compliance

| Finding | Severity | Why It Matters |
|---|---|---|
| Phase 6 of the original plan was a defensive guard inside `TryParseNavigationChain` keyed on `columnLookup`. The Phase 3 syntactic fallback at `ProjectionAnalyzer.cs:1939-1952` returns *before* `TryParseNavigationChain` is reached for any `o.X.Id` syntax, regardless of whether `X` is a column. The Decisions log calls this "subsumed" â€” but the original Phase 6 was scoped to the case where `X âˆˆ columnLookup`; the new fallback fires for `X âˆ‰ columnLookup` too. | In `ChainAnalyzer.cs:2161-2193`, when `IsRefKeyAccess=true` but the registry lookup misses (`fkCol == null`), the code falls through to the generic enrichment block at line 2195. If the generic block then matches `entityCol` for some other reason (e.g., a registry that has a non-FK column whose property name happens to equal `col.ColumnName` â€” the FK property name in the placeholder), line 2238 sets `IsForeignKey = entityCol.Kind == ColumnKind.ForeignKey`. That state is internally inconsistent and currently unreachable in practice (no test exercises it), but it is a latent invariant violation worth documenting or asserting on.
