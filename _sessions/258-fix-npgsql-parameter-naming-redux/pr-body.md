## Summary

- Closes #258
- Actually fixes the Npgsql 10 `08P01: bind message supplies 0 parameters` reported against v0.3.0 and still reproducing on v0.3.1 / v0.3.2 after PR #261.
- Root cause (proven empirically with an in-repo probe): Npgsql switches between named and positional binding modes based on whether any `DbParameter` has a `ParameterName` set — not based on what `CommandText` contains. PR #261's theory that "Npgsql 10 strictly matches parameters by name" was wrong. Both v0.3.0 (`$N` SQL + `@pN` names) and v0.3.1/v0.3.2 (`$N` SQL + `$N` names) flip Npgsql into named-lookup mode against SQL that has no `@name` markers, and the Bind frame ships with zero values.
- Fix: keep the `$N` placeholders in PG CommandText (Quarry already emits those) and leave `DbParameter.ParameterName` empty on PostgreSQL so Npgsql stays on its native positional-binding path.

## Reason for Change

The customer-facing symptom (via decompile of the shipped v0.3.2 binary) was `InsertHistoryRowAsync` returning `08P01` and failing 100% of integration tests on a Testcontainers-backed PG 17 setup. PR #261 tried to fix this by aligning `ParameterName` to the `$N` placeholders — which produces the "C" configuration that turns out to fail identically. In addition, the same broken pattern was applied in three more generator paths (entity insert, batch insert, collection expansion), so `MigrationRunner` was not the only affected code path — any chain-API `Insert`, `InsertBatch`, or non-constant `WHERE IN (collection)` on PG would have reproduced the bug.

## Impact

- `MigrationRunner.RunAsync` now succeeds on Npgsql 10 + PG 17. The originally reported #258 scenario works end-to-end.
- Generated chain-API code on PostgreSQL (`Insert(entity)`, `InsertBatch`, `WHERE IN (collection)`) now binds parameters correctly under Npgsql 10's strict binding.
- `QueryTestHarness.Pg` moves off `MockDbConnection` onto a real `NpgsqlConnection` backed by a shared Testcontainers PG 17 container — "free" PG execution coverage for any existing test that wants to flip from `Lite.ExecuteXxxAsync` to `Pg.ExecuteXxxAsync`.
- **(Phase 9 — session 2)** Every existing `CrossDialect*Tests` test that already executed on Lite now ALSO executes on real Pg (~233 mirror blocks across 22 files). This closes the test-coverage gap that hid PR #261's regression: cross-dialect tests previously only verified SQL-string shape on Pg, never executed through the real provider.
- Developers and CI runners need Docker for the test suite. Developers without Docker get a single clear "Install Docker" Ignored result instead of cascading exceptions (`PostgresTestContainer.GetContainerAsync` catches Docker-unavailable errors and calls `Assert.Ignore`).
- **Consumer rebuild requirement:** the fix is in the Quarry source generator. NuGet consumers MUST rebuild their consuming project after upgrading Quarry — the generated interceptor `__pN.ParameterName` assignments change from `"$N"` (or `ParameterNames.Dollar(...)`) to `""`. A binary-only package bump without a rebuild will not pick up the fix.

## Gaps in original plan implemented

- **MigrationRunner DateTime bug surfaced by Phase 5 testing.** `InsertHistoryRowAsync` passed `DateTime.UtcNow.ToString("o")` (a string) for `applied_at` / `started_at`. SQLite's TEXT column tolerated this; PG's `TIMESTAMP NOT NULL` rejected with `42804: column "applied_at" is of type timestamp without time zone but expression is of type text`. Fix: bind `DateTime` directly. Subtle behavior change: `applied_at` and `started_at` now share one captured `now` timestamp per invocation (arguably more correct for a migration batch). SQLite unaffected.
- **Review #4 (critical, surfaced by review phase).** The same `$N`-ParameterName bug existed in a fifth code path that my original Phase 3 work missed: `CarrierEmitter.cs:690` assigns `__pc.ParameterName = __col{i}Parts[__bi]` inside the runtime-collection-expansion loop, and `__colNParts` holds `$N` strings for SQL text. Any `.Contains(nonConstCollection)` on PG would reproduce the v0.3.1/v0.3.2 failure. My original `WhereInCollection` integration test didn't catch it because `new[] { 1, 3 }` is constant-folded to literal SQL. Fix: CarrierEmitter now branches on dialect in the collection loop (PG → `""`, others → `__colNParts[__bi]`). The integration test now sources the array via a helper method so SqlExprAnnotator cannot inline it; verified by inspecting the generated interceptor that the `__col0Len` runtime loop IS emitted. Added two new `CarrierGenerationTests` source-level guards (`CarrierGeneration_WhereInCollection_*`).
- **Harness resource leak on partial-setup failure (review #5).** `QueryTestHarness.CreateAsync` was not wrapped in try/catch; a transient mid-setup exception would leak Sqlite/Npgsql/Mock connections until process exit. Added unwind code.
- **Docker-unavailable fallback (review #25).** `PostgresTestContainer.GetContainerAsync` now detects Docker unavailability (type-name / message heuristic) and `Assert.Ignore`s all PG-backed tests with a clear "Install Docker" message instead of surfacing a cascade of broken tests.
- **Concurrent-process-safe baseline (review #20).** `EnsureBaselineAsync` replaced drop-and-recreate with PG advisory-lock-gated idempotent DDL. Two concurrent NUnit processes sharing one container no longer stomp on each other's baseline schema.
- **Migration test teardown logging (review #6).** `PostgresMigrationRunnerTests.TearDown` now writes `DROP SCHEMA` failures to `TestContext.Out` instead of swallowing silently, so orphan-schema accumulation in long-running dev containers is diagnosable.
- **Class visibility consistency (review #16).** `PostgresIntegrationTests` flipped from `internal class` to `public class` to match the two sibling PG test fixtures.
- **(Phase 9) PG container DDL alignment with `Quarry.Migration.SqlTypeMapper`.** Phase 4's initial DDL port from SQLite used `REAL`/`INTEGER`/`TEXT` for everything non-key, which Npgsql's strict typing rejected once Phase 9 actually executed parameterised queries against real columns. Updated `PostgresTestContainer.CreateSchemaObjectsAsync`:
  - `Col<decimal>` / `Col<Money>`→decimal: `DOUBLE PRECISION` → `NUMERIC(18, 2)` (decimals can't materialise from binary float).
  - `Col<bool>`: `INTEGER` → `BOOLEAN` (Quarry emits `WHERE IsActive = TRUE` / `bool` parameter binding on PG).
  - `Col<DateTime>`: `TEXT` → `TIMESTAMP`. `Col<DateTimeOffset>`: `TEXT` → `TIMESTAMPTZ`.
  - `FOREIGN KEY` clauses dropped — SQLite source declares them but doesn't enforce them by default; replicating in PG broke `Delete users` tests that rely on no FK enforcement. Mirrors Lite's effective semantics.
- **(Phase 9) Three PG-vs-SQLite divergences triaged inline:**
  - `Select_Distinct`: PG does not guarantee insertion-order return without ORDER BY; sort `pgResults` by `UserId` before assertions.
  - `Join_Distinct_OrderBy_Limit`: PG rejects DISTINCT with ORDER BY on a non-projected column (`42P10`); SQLite tolerates. Pg execution intentionally not mirrored — SQL-text assertion still verifies the generator emits the same SQL on both dialects.
  - `Cte_TwoChainedWiths_DistinctDtos_CapturedParams`: workaround for an unrelated Quarry source-generator bug in chained-With dispatch (closure-shape merging picks the first encountered variable name as the canonical extractor field, causing `MissingFieldException` at `.Prepare()` time when a different test in the same compilation unit used a different name). Renamed `orderCutoff` → `cutoff` so the closure shape matches the expected chain. Underlying generator bug is independent of #258 and to be filed as a follow-up.

## Migration Steps

None. Consumer-facing public API is unchanged. `RawSqlAsync` contract (`@pN` placeholders + `@pN` names) is unchanged. The only consumer action required is **rebuilding** the consuming project after upgrading Quarry — the source generator re-emits interceptors, and binary-only package bumps will not pick up the fix.

## Performance Considerations

- Per-command: Npgsql now stays on its native positional-binding path on PG (no `@name`→`$N` rewrite pass, no name-to-index map construction). Strictly less work than the working-but-suboptimal "A" configuration (`@pN` SQL + `@pN` names).
- Test suite: +2s across ~3000 tests for the real-Npgsql harness upgrade (pool-backed connection acquire + `SET search_path` + `BEGIN` + `ROLLBACK` per test). Phase 9's mirror added ~233 additional Pg `ExecuteXxxAsync` invocations; total wall-clock for `Quarry.Tests` remains ~20s.

## Security Considerations

- No new attack surface. All changes operate on `DbParameter.ParameterName` metadata and test-only infrastructure. No user-supplied data newly flows into the parameter layer or into DDL (test-harness schema names are Guid-derived).
- New dev-only dependency `Testcontainers.PostgreSql 4.*` in `Quarry.Tests.csproj`. Not shipped to consumers.

## Breaking Changes

**Consumer-facing:** None. Public API unchanged. Users on Npgsql 9 and on non-PG providers are unaffected. Users on Npgsql 10 + PG get the bug fix. Must rebuild after upgrading.

**Internal:** `CarrierEmitter.FormatParamName` / `EmitParamNameExpr` visibility unchanged from PR #261 (`internal static`). `QueryTestHarness` gains a new `useOwnPgSchema` parameter with a sensible default; no call-site update needed. Phase 9 test-only edits do not affect any production-code-facing API.
