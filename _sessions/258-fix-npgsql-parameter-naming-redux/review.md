## Correctness

| # | Finding | Severity | Why It Matters |
|---|---------|----------|----------------|
| 4 | Collection IN-clause path (`CarrierEmitter.cs:690`) still assigns `__pc.ParameterName = __col{i}Parts[__bi]`, and for PG `__colNParts` is populated with `"$N"` strings via `TerminalEmitHelpers.EmitCollectionPartsPopulation` (line 910). This is the exact v0.3.1/v0.3.2 "C" failure configuration (`$N` SQL + `$N` names) that the probe proves fails with 08P01 | critical | If a chain-API query on PG uses a `.Contains(nonConstCollection)` pattern where the collection is a local variable / method parameter (not a compile-time const), the generator emits runtime-expanded `$N` placeholders AND assigns the same `$N` strings as `ParameterName` — flipping Npgsql into named-lookup mode and reproducing the v0.3.1 bug. `ParameterNames.Dollar` was correctly kept for SQL-text assembly, but its reuse as `ParameterName` source at `CarrierEmitter.cs:690` was not updated. The new PG integration test (`WhereInCollection_OnPostgreSQL_ExecutesSuccessfully`) does NOT cover this path — `new[] { 1, 3 }` gets constant-folded to `IN (1, 3)` literals (verified by the manifest entry `WHERE "UserId" IN (1, 3)`). Needs a test with a non-constant collection parameter on PG, and the fix: emit `""` for the ParameterName assignment at line 690, while keeping `$N` in the `__colNParts` array for SQL text. |
| 5 | Resource leak in `QueryTestHarness.CreateAsync` setup: if `BeginTransactionAsync`, `CreateOwnedSchemaAsync`, `SetSearchPathAsync`, `CreateSchema` or `SeedData` throws, the opened `NpgsqlConnection`, `SqliteConnection`, and `MockDbConnection` leak because creation is not wrapped in a try/catch that disposes on partial failure | medium | Test-setup exceptions (e.g. a transient PG connection refusal, a DDL mismatch) would permanently leak an open Npgsql pool connection per test until process exit. In a CI run with many failing tests this could exhaust the container's connection limit and cascade into further unrelated failures. Standard fix: wrap the body in try/catch that disposes partial state on throw. |
| 6 | `PostgresMigrationRunnerTests.TearDown` catches all exceptions from `DROP SCHEMA CASCADE` with an empty `catch { }`; silent drop failures could leave orphan schemas in the shared container across test runs | low | Only affects the long-running dev container; CI tears down the container at end of run. Acceptable for now, but worth logging to help diagnose flakes if schema-count ever grows unbounded. |
| 7 | DateTime fix in `MigrationRunner.InsertHistoryRowAsync` passes a single captured `now` for both `applied_at` and `started_at`, replacing two separate `DateTime.UtcNow` calls | low | Previously, `started_at` could be a few ticks after `applied_at`. Now they're identical. Semantically "this migration batch's wall-clock" is arguably more correct; the change is justified in context and worth calling out as a subtle behavior shift independent of the PG fix. |
| 8 | `PostgresTestContainer.GetContainerAsync` has a read-before-lock double-checked locking pattern without `Volatile.Read`/`volatile` — technically a weak-memory-model hazard on ARM, but practically fine on x64 Windows/Linux targets | info | Informational. No action needed unless Quarry targets ARM CI. |
