## Description

Extend the cross-dialect execution coverage that PR #266 (issue #258) added for PostgreSQL to MySQL. After PR #266, every `CrossDialect*Tests` test executes on real SQLite **and** real PostgreSQL, but `My` and `Ss` are still backed by `MockDbConnection` and only verify SQL-string shape. This means the regression class that motivated #258 (a generator-emitted parameter-binding pattern that produces broken SQL on the real provider while passing all mock-based tests) is **still uncovered for MySQL**.

This issue tracks adding the same end-to-end execution coverage for MySQL: real MySqlConnector against a Testcontainers-backed MySQL container, with every existing Lite-execute test mirrored to a My-execute test.

## Reference Implementation

PR #266 / issue #258 is the reference for this work. The same shape applies to MySQL with provider-specific substitutions:

| PostgreSQL surface | MySQL substitution |
|---|---|
| `Testcontainers.PostgreSql 4.*` package | `Testcontainers.MySql 4.*` package |
| `Npgsql 10.*` (already in csproj) | `MySqlConnector 2.*` (verify whether already referenced) |
| `src/Quarry.Tests/Integration/PostgresTestContainer.cs` | `src/Quarry.Tests/Integration/MySqlTestContainer.cs` (new) |
| `postgres:17-alpine` image | `mysql:8.4` (LTS) image |

| `Quarry.Migration.SqlTypeMapper.MapMySql` (already exists) | This is what the harness DDL must align with |
| `QueryTestHarness.Pg` real-connection wiring | `QueryTestHarness.My` parallel wiring |

## Scope

- Add `Testcontainers.MySql` to `src/Quarry.Tests/Quarry.Tests.csproj`. Verify `MySqlConnector` is referenced explicitly.
- Create `src/Quarry.Tests/Integration/MySqlTestContainer.cs` mirroring `PostgresTestContainer`:
  - Lazy, process-wide singleton container.
  - `EnsureBaselineAsync` populates a single `quarry_test` database with the test schema + seed data, gated by `GET_LOCK` for concurrent-process safety.
  - `CreateOwnedDatabaseAsync` for the per-test opt-out path (used by migration runner / transactional tests).
  - Docker-unavailable detection routed to `Assert.Ignore` with a clear "Install Docker" message (mirror the PG path).
- Port the schema DDL (`CreateSchemaObjectsAsync`) to MySQL using `Quarry.Migration.SqlTypeMapper.MapMySql` as the source of truth:
  - `Col<bool>` to `TINYINT(1)` (per `SqlTypeMapper.cs:86`).
  - `Col<decimal>` / `Col<Money>` to `DECIMAL(18,2)` (per `SqlTypeMapper.cs:90`).
  - `Col<DateTime>` to `DATETIME`, `Col<DateTimeOffset>` to `DATETIME` (per `SqlTypeMapper.cs:93-94` -- note: DTO maps to plain DATETIME on MySQL; DST/offset semantics may need triage).
  - `Col<Guid>` to `CHAR(36)`.
  - Auto-increment: `INT NOT NULL AUTO_INCREMENT PRIMARY KEY` instead of PG's `GENERATED BY DEFAULT AS IDENTITY`.
  - **Identifier quoting:** MySQL uses backticks. The schema port must NOT keep PG-style double-quoted identifiers.
  - **FK constraints:** the SQLite source declares them but doesn't enforce; PG port dropped them to mirror SQLite's effective semantics. MySQL with InnoDB does enforce FKs, so the same drop applies.
- Upgrade `QueryTestHarness.My` from `MockDbConnection` to a real `MySqlConnection` backed by the shared container. Default to transactional (`BEGIN`/`ROLLBACK`); add a `useOwnMyDatabase: true` opt-out for tests that need real commits.
- Mirror every existing `await lt.ExecuteXxxAsync(...)` block in `src/Quarry.Tests/SqlOutput/CrossDialect*Tests.cs` with a parallel `await my.ExecuteXxxAsync(...)` block applying the same assertions. Same mechanical pattern PR #266 used for Pg.

## Diagnostics / What to Watch For

When the mirror is applied, expect the same triage categories that PR #266 surfaced for PG:
- **Decimal / float type mismatches** -- MySQL is generally lenient but `DECIMAL` vs `REAL` should be checked against `reader.GetDecimal(i)`.
- **Boolean materialisation** -- MySQL stores BOOL as `TINYINT(1)`. Verify `reader.GetBoolean(i)` works correctly against TINYINT and that `WHERE col = TRUE` parameter binding is `1`/`0` or `bool` consistently.
- **DateTime / DateTimeOffset** -- MySQL `DATETIME` has no offset awareness. `Col<DateTimeOffset>` round-trip behaviour needs triage; possibly use `TIMESTAMP` for offset-aware columns.
- **Row-order without ORDER BY** -- MySQL InnoDB does not guarantee insertion order. Same fix pattern as `Select_Distinct` in PR #266: sort `myResults` before assertions where Lite happened to be insertion-ordered.
- **GROUP BY strictness** -- MySQL 8 enforces ONLY_FULL_GROUP_BY by default (similar to the PG 42P10 issue but on a different shape). Tests that select non-aggregated non-grouped columns may fail and require either generator changes or test rewrites.
- **Identifier case sensitivity** -- depends on `lower_case_table_names` server config. Containerised MySQL on Linux defaults to case-sensitive (1); on Windows host volumes, things can drift. Pin the container config explicitly.
- **Latent generator bugs hidden by mocks** -- like the chained-With dispatch bug (#268) that PR #266 surfaced for PG, MySQL execution may expose its own latent issues. Triage each on its own merits.

## Gathered Information

- `Quarry.Migration.SqlTypeMapper.MapMySql` (lines 78-100 of `src/Quarry/Migration/SqlTypeMapper.cs`) is already the production CLR-to-SQL type mapping for MySQL and serves as the authoritative reference for what column types the harness DDL must use.
- `MockDbConnection` is currently used for both `My` and `Ss` in `QueryTestHarness`; the wiring point is the `My` field assignment in `QueryTestHarness.CreateAsync`. PR #266's PG migration is the literal template -- diff `QueryTestHarness.cs` between `master~10` and `master` for the exact change shape.
- `quarry-manifest.mysql.md` will regenerate automatically once tests start executing on the real provider -- review it after the mirror lands to confirm the manifest reflects the expanded surface, similar to `quarry-manifest.postgresql.md` after PR #266.
- CI runners on `ubuntu-latest` already have Docker and ran the PG container; MySQL will reuse the same Docker daemon. No CI infra change needed beyond the additional container startup time.
- A linked tracking issue, **#268**, may need to be resolved before the MySQL mirror can land cleanly -- if the chained-With dispatch bug fires on `My` for the same compilation state, the same workaround (or the underlying fix) must be applied.

## Suggested Approach

Phase the work the same way PR #266 did:
1. **Phase 1 -- container helper:** add `Testcontainers.MySql`, build `MySqlTestContainer.cs`, single regression-probe test confirming Docker bootstrap works on CI.
2. **Phase 2 -- DDL port + harness upgrade:** port the schema DDL to MySQL aligned with `SqlTypeMapper.MapMySql`; flip `QueryTestHarness.My` to a real connection.
3. **Phase 3 -- focused integration tests:** four mirrors of the Phase-5 PG tests (entity insert, batch insert, where-in-collection, migration runner) on My.
4. **Phase 4 -- full cross-dialect mirror:** delegate the mechanical mirror across the 22 CrossDialect files (same agent pattern PR #266 used).
5. **Phase 5 -- triage:** classify each My-only failure as (a) real bug to fix, (b) MySQL-specific behaviour to work around with inline comment, (c) latent generator bug worth filing as a follow-up issue.

Land as a single PR if the diff stays under ~3000 net additions; split into "harness + integration tests" then "full mirror" if larger.

## Out of Scope

- Multi-version MySQL coverage (5.7, 8.0, 8.4). Pick MySQL 8.4 LTS for this issue.
- MariaDB compatibility. Tracked separately if/when it arises.
- PG-only / SQLite-only test triage that has nothing to do with MySQL.

PR #266 / issue #258, while shipping PG execution coverage. The same "Pg used MockDbConnection so generator regressions could ship undetected" risk applies symmetrically to My and Ss. Issue #258 is the cautionary tale.
