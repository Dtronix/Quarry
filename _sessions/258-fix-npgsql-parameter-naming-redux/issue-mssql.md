## Description

Extend the cross-dialect execution coverage that PR #266 (issue #258) added for PostgreSQL to SQL Server. After PR #266, every `CrossDialect*Tests` test executes on real SQLite **and** real PostgreSQL, but `My` and `Ss` are still backed by `MockDbConnection` and only verify SQL-string shape. This means the regression class that motivated #258 (a generator-emitted parameter-binding pattern that produces broken SQL on the real provider while passing all mock-based tests) is **still uncovered for SQL Server**.

This issue tracks adding the same end-to-end execution coverage for SQL Server: real Microsoft.Data.SqlClient against a Testcontainers-backed MS SQL container, with every existing Lite-execute test mirrored to an Ss-execute test.

PR #266 / issue #258 is the reference for this work. The same shape applies to SQL Server with provider-specific substitutions:

## Scope

- Add `Testcontainers.MsSql` to `src/Quarry.Tests/Quarry.Tests.csproj`. Verify `Microsoft.Data.SqlClient` is referenced explicitly.
- Create `src/Quarry.Tests/Integration/MsSqlTestContainer.cs` mirroring `PostgresTestContainer`:
  - Lazy, process-wide singleton container.
  - `EnsureBaselineAsync` populates a single `quarry_test` schema with the test tables + seed data, gated by `sp_getapplock` for concurrent-process safety.
  - `CreateOwnedSchemaAsync` for the per-test opt-out path (used by migration runner / transactional tests).
  - Docker-unavailable detection routed to `Assert.Ignore` with a clear "Install Docker" message (mirror the PG path).
  - Note: MS SQL container takes longer to bootstrap than PG (cold start ~20-30s vs ~5s) and requires accepting the EULA via env var. Plan for the additional CI time.
- Port the schema DDL (`CreateSchemaObjectsAsync`) to SQL Server using `Quarry.Migration.SqlTypeMapper.MapSqlServer` as the source of truth:
  - `Col<bool>` to `BIT` (per `SqlTypeMapper.cs:110`).
  - `Col<decimal>` / `Col<Money>` to `DECIMAL(18,2)` (per `SqlTypeMapper.cs:114`).
  - `Col<DateTime>` to `DATETIME2`, `Col<DateTimeOffset>` to `DATETIMEOFFSET` (per `SqlTypeMapper.cs:117-118`).
  - `Col<Guid>` to `UNIQUEIDENTIFIER`.
  - Auto-increment: `INT IDENTITY(1,1) PRIMARY KEY` instead of PG's `GENERATED BY DEFAULT AS IDENTITY`. Note: SQL Server IDENTITY does not allow explicit value insertion by default; use `SET IDENTITY_INSERT <table> ON` around seed inserts (mirroring PG's identity-reseed pattern at the end of seeding).
  - **Identifier quoting:** SQL Server uses square brackets. The schema port must NOT keep PG-style double-quoted identifiers.
  - **FK constraints:** the SQLite source declares them but doesn't enforce; PG port dropped them to mirror SQLite's effective semantics. SQL Server enforces FKs by default, so the same drop applies.
- Upgrade `QueryTestHarness.Ss` from `MockDbConnection` to a real `SqlConnection` backed by the shared container. Default to transactional (`BEGIN TRANSACTION` / `ROLLBACK TRANSACTION`); add a `useOwnSsSchema: true` opt-out for tests that need real commits.
- Mirror every existing `await lt.ExecuteXxxAsync(...)` block in `src/Quarry.Tests/SqlOutput/CrossDialect*Tests.cs` with a parallel `await ss.ExecuteXxxAsync(...)` block applying the same assertions. Same mechanical pattern PR #266 used for Pg.

## Diagnostics / What to Watch For

When the mirror is applied, expect the same triage categories that PR #266 surfaced for PG, plus SQL-Server-specific concerns:
- **Decimal precision** -- SQL Server is strict about `DECIMAL` precision/scale on parameter binding. Verify Quarry's parameter type inference matches the column declared scale.
- **Boolean materialisation** -- SQL Server `BIT` is 0/1; `reader.GetBoolean(i)` should work but `WHERE col = 1` vs `WHERE col = TRUE` is a generator concern -- check `Quarry.Migration.SqlTypeMapper.MapSqlServer` and the corresponding SQL emission for boolean comparisons.
- **DateTime / DateTimeOffset** -- SQL Server has both `DATETIME2` (offset-naive) and `DATETIMEOFFSET` (offset-aware). The mapping above is correct per SqlTypeMapper, but parameter binding for `Col<DateTime?>` against a `DATETIME2 NULL` column should be checked against `SqlDbType` defaults.
- **Row-order without ORDER BY** -- SQL Server makes no insertion-order guarantee. Same fix pattern as `Select_Distinct` in PR #266: sort `ssResults` before assertions where Lite happened to be insertion-ordered.
- **OFFSET/FETCH dialect specifics** -- SQL Server uses `OFFSET ... ROWS FETCH NEXT ... ROWS ONLY` instead of `LIMIT`. The current cross-dialect tests already assert this in their `ss:` SQL strings, so SQL generation is exercised; the new question is whether the executed query returns the expected rows on real SQL Server.
- **`SELECT TOP` vs `OFFSET/FETCH`** -- TOP is not valid in subqueries with ORDER BY without OFFSET. Triage if any test fails on this shape.
- **Identifier case sensitivity** -- depends on the database collation. SQL Server containers default to case-insensitive. Quarry's quoted identifiers should still resolve correctly, but verify on real container.
- **Latent generator bugs hidden by mocks** -- like the chained-With dispatch bug (#268) that PR #266 surfaced for PG, SQL Server execution may expose its own latent issues. Triage each on its own merits.

## Gathered Information

- `Quarry.Migration.SqlTypeMapper.MapSqlServer` (lines 102-124 of `src/Quarry/Migration/SqlTypeMapper.cs`) is already the production CLR-to-SQL type mapping for SQL Server and serves as the authoritative reference for what column types the harness DDL must use.
- `MockDbConnection` is currently used for both `My` and `Ss` in `QueryTestHarness`; the wiring point is the `Ss` field assignment in `QueryTestHarness.CreateAsync`. PR #266's PG migration is the literal template -- diff `QueryTestHarness.cs` between `master~10` and `master` for the exact change shape.
- `quarry-manifest.sqlserver.md` will regenerate automatically once tests start executing on the real provider -- review it after the mirror lands to confirm the manifest reflects the expanded surface, similar to `quarry-manifest.postgresql.md` after PR #266.
- CI runners on `ubuntu-latest` already have Docker and ran the PG container; the MS SQL container is heavier than PG. Plan for ~30s extra cold-start cost in CI runs.
- The MS SQL Linux container requires `ACCEPT_EULA=Y` and a strong `MSSQL_SA_PASSWORD` env var. `Testcontainers.MsSql` builder handles this; verify the password meets the complexity requirements.
- A linked tracking issue, **#268**, may need to be resolved before the SQL Server mirror can land cleanly -- if the chained-With dispatch bug fires on `Ss` for the same compilation state, the same workaround (or the underlying fix) must be applied.

## Suggested Approach

Phase the work the same way PR #266 did:
1. **Phase 1 -- container helper:** add `Testcontainers.MsSql`, build `MsSqlTestContainer.cs`, single regression-probe test confirming Docker bootstrap works on CI (with extra warmup tolerance).
2. **Phase 2 -- DDL port + harness upgrade:** port the schema DDL to SQL Server aligned with `SqlTypeMapper.MapSqlServer`; flip `QueryTestHarness.Ss` to a real connection. Plan for `IDENTITY_INSERT` toggling around seed.
3. **Phase 3 -- focused integration tests:** four mirrors of the Phase-5 PG tests (entity insert, batch insert, where-in-collection, migration runner) on Ss.
4. **Phase 4 -- full cross-dialect mirror:** delegate the mechanical mirror across the 22 CrossDialect files (same agent pattern PR #266 used).
5. **Phase 5 -- triage:** classify each Ss-only failure as (a) real bug to fix, (b) SQL-Server-specific behaviour to work around with inline comment, (c) latent generator bug worth filing as a follow-up issue.

Land as a single PR if the diff stays under ~3000 net additions; split into "harness + integration tests" then "full mirror" if larger.

## Out of Scope

- Multi-version SQL Server coverage (2017, 2019, 2022). Pick 2022-latest for this issue.
- Azure SQL DB / SQL Server on Windows. Linux container only.
- LocalDB or full SQL Server Express. Not required for parity.
- PG-only / SQLite-only / MySQL-only test triage that has nothing to do with SQL Server.

## Surfaced By

PR #266 / issue #258, while shipping PG execution coverage. The same "Pg used MockDbConnection so generator regressions could ship undetected" risk applies symmetrically to My and Ss. Issue #258 is the cautionary tale.
